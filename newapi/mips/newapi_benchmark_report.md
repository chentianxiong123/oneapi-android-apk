# 第一章：设备绝对性能

本章数据在**同一局域网内有线连接**下测得，目的是比较设备 CPU 和软件栈的极限吞吐能力。

## 三设备对比分析

### 主表：常规性能 (无前端 · 亮屏)

| 并发 | HaiNaSi RPS | Xiaomi 23049RAD8C RPS | Xiaomi 23049RAD8C USB RPS | Xiaomi M5 Note RPS | Huawei RPS |
|-----:|:----------:|:--------------------:|:------------------------:|:-----------------:|:----------:|
| 5 | - | 393 | - | - | - |
| 10 | - | 773 | **1297** | 476 | 300 |
| 20 | - | 614 | - | - | - |
| **50** | **585** | **997** | 675 | **935** | **637** |
| **100** | **436** | 597 | 493 | **1098** | **618** |
| **200** | 275 | 459 | 450 | **1132** | 576 |

> 全部 0 错误，全部 Wi-Fi。HaiNaSi 缺少 5/10/20 无前端数据。

### 关键发现

| 维度 | 现象 | 原因 |
|------|------|------|
| **低并发 (<20)** | Xiaomi 23049RAD8C 领先（795/s） | 2.3GHz 大核单线程碾压 |
| **中并发 (50)** | 23049RAD8C 与 M5 Note 持平（964 vs 935） | 8核全开不相上下 |
| **高并发 (100+)** | M5 Note 领先逼近 1100/s | WiFi 网卡更好，不拖后腿 |
| **海纳思** | 四台中最弱 | 4×A53 @ 1.5GHz 上限低 |
| **Xiaomi 23049RAD8C** | 50→100 线程 RPS 降 38% | WiFi 网卡高并发饱和 |

### 瓶颈汇总

| 设备 | 瓶颈所在 | 拐点 | 峰值 RPS |
|------|---------|:----:|:--------:|
| **HaiNaSi** (WiFi) | 🔴 CPU（4×A53 @ 1.5GHz 满载） | 42线程 | 585 |
| **Xiaomi 23049RAD8C** (WiFi) | ⚠️ WiFi 饱和后 CPU 有余 | 50线程 | **997** |
| **Xiaomi 23049RAD8C** (USB RNDIS) | ⚠️ USB 协议瓶颈 | 10线程 | **1297** |
| **Xiaomi M5 Note** (WiFi) | ⚠️ 接近 CPU 饱和 | ~150线程 | **1132** |
| **Huawei** (WiFi) | 🔴 CPU + WiFi 混合 | 50线程 | 637 |

**结论：** 在网络条件相同时（有线），CPU 决定上限；WiFi 场景下，手机网卡的并发处理能力可能先于 CPU 成为瓶颈。Huawei 数据为亮屏。

## 瓶颈诊断

### HaiNaSi (vmstat 实测)

| 指标 | 空闲 | 50线程峰值 | 结论 |
|------|:---:|:---------:|------|
| CPU 用户态 | 1% | 60% | 🔴 应用处理瓶颈 |
| CPU 内核态 | 1% | 40% | 🔴 上下文切换密集 |
| CPU 空闲 | 99% | 4-8% | 🔴 CPU 耗尽 |
| IO 等待 | 0% | 0% | ✅ 磁盘非瓶颈 |

**结论：HaiNaSi 瓶颈在 CPU。** 4×A53 1.5GHz，50线程最优 585/s。

### Xiaomi 23049RAD8C

| 并发 | WiFi RPS | USB RNDIS RPS | 延迟(WiFi) | 延迟(USB) | 现象 |
|-----:|:--------:|:-------------:|:----------:|:---------:|------|
| 5 | 393 | - | 12ms | - | CPU 远未饱和 |
| **10** | 773 | **1297** | 11ms | **7ms** | USB 直连快 68%，WiFi 开销消失 |
| 20 | 614 | - | 29ms | - | 开始受 WiFi 影响 |
| **50** | **997** | 675 | **43ms** | 64ms | WiFi 强于 USB RNDIS |
| 100 | 597 | 493 | 144ms | 176ms | WiFi 仍在 RNDIS 之上 |
| 200 | 459 | 450 | 373ms | 383ms | 两者拉平 |

**USB RNDIS vs WiFi：**
- 低并发（10线程）USB RNDIS 大幅领先（1297 vs 773），因为绕过了 WiFi 协议栈开销
- 高并发（50+）WiFi 反超。RNDIS 是 USB 2.0 单队列协议，并发能力有限
- 10线程 1297/s 是这台手机**纯 CPU 性能的下限**—连网络开销都减到最低还能跑这么多

**结论：** 要真正测满 23049RAD8C 的 CPU，需要 USB-C 转以太网或有线连接。RNDIS 在低并发下表现亮眼，但高并发受 USB 协议限制。

### Xiaomi M5 Note

| 并发 | WiFi RPS | WiFi 延迟 | ADB USB RPS | ADB USB 延迟 |
|-----:|:--------:|:--------:|:-----------:|:------------:|
| 10 | 476 | 19ms | **722** | 12ms |
| **50** | **935** | 46ms | 393 | 111ms |
| **100** | **1098** | 78ms | 347 | 249ms |
| 200 | **1132** | 149ms | 318 | 544ms |

**WiFi vs ADB USB 对比：**
- 低并发 WiFi 比 ADB USB 慢 34%（476 vs 722），WiFi 延迟开销正常
- 高并发 ADB USB 成为瓶颈，吞吐量反而下降
- WiFi 下吞吐量随并发稳定增长，峰值 **1132/s**

**结论：M5 Note WiFi 网卡素质好，高并发不拉胯。** 骁龙 625 的 8×A53 @ 2.0GHz 配合均衡的 WiFi 硬件，100+ 线程后依然保持 1000+ RPS。

### Huawei RIO-UL00

| 并发 | APK(亮屏) | APK(息屏) | 纯二进制(ADB) | 亮屏延迟 | 息屏延迟 | 纯二进制延迟 |
|-----:|:--------:|:--------:|:------------:|:--------:|:--------:|:----------:|
| 10 | 300 | 252 | 255 | 30ms | 36ms | 36ms |
| **50** | **637** | 468 | 471 | **68ms** | 93ms | 92ms |
| **100** | **618** | 504 | **553** | 140ms | 164ms | 152ms |
| 200 | 576 | 470 | 527 | 302ms | 358ms | 323ms |

**APK vs 纯二进制：** APK 前台服务比 ADB shell 快 10-26%。原因是 Android 进程优先级不同。

### 华为 CPU 架构分析

华为 RIO-UL00 搭载 **骁龙 616**（非麒麟），big.LITTLE HMP 架构：

| 集群 | 核心 | 频率 | 说明 |
|------|:----:|:----:|------|
| cluster 0 (cpu0-3) | 4×Cortex A53 | **1.5GHz** | 大核簇 |
| cluster 4 (cpu7-4) | 4×Cortex A53 | **1.2GHz** | 小核簇 |

Android 通过 cpuset cgroup 限制不同优先级进程可用的 CPU 核心：

| 运行方式 | cpuset | 可用核心 | 表现 |
|---------|:------:|:--------:|:----:|
| **APK 前台服务** | `top-app` / `foreground` | 全核（含 1.5GHz 大核） | **637/s 峰值** |
| **APK 息屏** | `background` | 仅 1.2GHz 小核 + 系统降频 | 468/s，降 26% |
| **纯二进制(ADB shell)** | `background` | 仅 1.2GHz 小核 | 471/s，降 26% |

**结论：** 纯二进制跑在 ADB shell 下 = 息屏级性能。CPU 频率限制 (1.2/1.5 = 80%) 与实测差距 (637→471 = 74%) 基本吻合，其余损耗来自调度优先级和 cgroup 时间片分配。

**结论：瓶颈不在 CPU。** 8核 + 15GB 但 50线程输给 4×A53，原因：
- 手机 WiFi 网卡比机顶盒网卡更省电优先、并发小包处理弱
- 有线对比有线才能测出真实 CPU 差距

# 第二章：真实场景可行性分析

## 场景：异地组网 P2P 分享

你和朋友各自家里放一台吃灰设备跑 NewAPI，通过 EasyTier / Tailscale / WireGuard 打通局域网。

## 瓶颈在哪？

```
请求链路：朋友A → 宽带A上行 → 互联网 → 宽带B下行 → 你的设备 → SQLite → 响应
                                      ↓
                               真正的瓶颈在这里
```

| 环节 | 典型值 | 是否瓶颈 |
|------|:-----:|:--------:|
| 设备 CPU 处理请求 | **<1ms** | ❌ |
| 设备局域网延迟 | 1-3ms | ❌ |
| **家庭宽带上行带宽** | **10-50 Mbps** | **✅ 是** |
| P2P 隧道加密 | 1-5ms 增加 | ⚠️ 可接受 |
| **跨运营商延迟** | **10-80ms** | **⚠️ 看命** |

## 设备性能够用吗？

用每天最极端的情况估算：**10 个朋友，每人每 5 秒发一次请求。**

```
10 人 × (60s / 5s) × 60min × 24h = 172,800 次/天
```

HaiNaSi 干这个只要 **172,800 ÷ 585/s = 295 秒 ≈ 5 分钟**。一天里有 **23 小时 55 分钟在空转。**

> **结论：任何一台 arm 设备都绰绰有余。**

## 真正的限制

```
家庭宽带上行 30 Mbps
每个流式响应 ~2MB（纯文本）
172,800 次/天 × 2MB = 345 GB/天 → 需要 32 Mbps 持续上行  
```

所以天花板是 **各家宽带上行带宽**，不是设备。

## 最终结论

| 维度 | 结论 |
|------|------|
| **设备性能排名** | 23049RAD8C USB(1297/s) > M5 Note(1132/s) > 23049RAD8C WiFi(997/s) > Huawei(637/s) > HaiNaSi(585/s) |
| **WiFi 影响** | 23049RAD8C USB 直连比 WiFi 快 68%，WiFi 砍掉近半吞吐 |
| **RNDIS 特性** | 低并发极快，高并发受 USB 2.0 单队列协议限制 |
| **部署方式** | APK 前台服务性能最佳，ADB shell 因 cpuset 限制慢 10-26% |
| **跨平台** | linux/arm + android/arm64 同一份代码 |
| **真正瓶颈** | 家庭宽带上行 (10-50 Mbps) |
| **P2P 方案** | EasyTier / Tailscale 零配置，延迟加几 ms |
| **前端/数据库开销** | 可忽略 |
